# Spec: memory-management

Change: feature-domain-knowledge-layer
Date: 2026-03-03

## Requirements

### Requirement: memory-init feature discovery step

`memory-init` MUST gain a feature discovery step that generates `ai-context/features/` stubs for detected bounded contexts when the directory does not yet exist.

The feature discovery step:
- MUST only execute when `ai-context/features/` does NOT exist (idempotent — never overwrites an existing directory)
- MUST attempt to detect bounded contexts from two sources:
  1. `openspec/specs/` subdirectory names (each subdirectory is a candidate domain slug)
  2. Primary source code directory names (`src/`, `app/`, `lib/`, or equivalent) — top-level subdirectory names are candidate domain slugs
- MAY use both sources; duplicate slugs MUST be deduplicated
- MUST generate one stub `.md` file per detected domain slug using the template at `ai-context/features/_template.md`
- If `_template.md` does not exist, the step MUST generate stubs using the canonical six-section structure inline (template absence MUST NOT block stub generation)
- Each generated stub MUST include a header comment: `<!-- Auto-generated by memory-init on [YYYY-MM-DD]. Fill in each section with actual domain knowledge. -->`
- MUST create `ai-context/features/_template.md` even if no bounded contexts are detected
- MUST NOT generate a stub for `_template.md` itself

#### Scenario: Feature discovery generates stubs from openspec/specs/

- **GIVEN** `ai-context/features/` does not exist
- **AND** `openspec/specs/` contains subdirectories: `auth`, `payments`, `notifications`
- **WHEN** `memory-init` runs the feature discovery step
- **THEN** `ai-context/features/` is created
- **AND** stubs are generated: `ai-context/features/auth.md`, `ai-context/features/payments.md`, `ai-context/features/notifications.md`
- **AND** `ai-context/features/_template.md` is created
- **AND** each stub contains the six-section structure with placeholder text

#### Scenario: Feature discovery generates stubs from src/ directory structure

- **GIVEN** `ai-context/features/` does not exist
- **AND** `openspec/specs/` is absent or empty
- **AND** the project has `src/auth/`, `src/payments/`, `src/shared/` directories
- **WHEN** `memory-init` runs the feature discovery step
- **THEN** stubs are generated for `auth` and `payments`
- **AND** `shared` MAY be skipped if the agent determines it is a utility/cross-cutting concern rather than a bounded context (this decision is the agent's discretion)

#### Scenario: Feature discovery is skipped when directory already exists

- **GIVEN** `ai-context/features/` already exists (with or without files)
- **WHEN** `memory-init` runs
- **THEN** the feature discovery step is skipped entirely
- **AND** no files in `ai-context/features/` are created, modified, or deleted
- **AND** `memory-init` proceeds to generate the other ai-context/ files normally

#### Scenario: Feature discovery runs with no detectable bounded contexts

- **GIVEN** `ai-context/features/` does not exist
- **AND** neither `openspec/specs/` nor a recognizable source directory structure is present
- **WHEN** `memory-init` runs the feature discovery step
- **THEN** `ai-context/features/` is created containing only `_template.md`
- **AND** no stub files are generated
- **AND** `memory-init` proceeds normally; this is not an error condition

#### Scenario: memory-init summary reports feature discovery outcome

- **GIVEN** `memory-init` creates `ai-context/features/` with three stubs
- **WHEN** `memory-init` returns its summary to the user
- **THEN** the summary MUST list the number of feature stubs generated
- **AND** remind the user that stubs contain placeholder text and should be filled in with actual domain knowledge

### Requirement: memory-update feature file update path

`memory-update` MUST gain a feature file update path so that session-acquired domain knowledge can be persisted to `ai-context/features/<domain>.md`.

The update path:
- MUST execute as part of the existing `memory-update` step sequence (after Step 6 changelog update, as a new Step 6b or integrated into Step 2's file determination logic)
- MUST apply when: a domain with an existing feature file was directly modified during the session (code changes, decision records, discovered invariants, resolved gotchas)
- MUST NOT create new feature files — only update files that already exist in `ai-context/features/`
- MUST write updates as append-or-merge operations: new rules added to Business Rules, new decisions appended to Decision Log, new gotchas appended to Known Gotchas — never overwriting existing content
- MUST respect `[auto-updated]` section boundary markers if present in feature files (consistent with existing `conventions.md` handling)
- If no session content is relevant to any feature file, this step MUST be skipped silently

#### Scenario: Session resolves a known gotcha — update is persisted

- **GIVEN** `ai-context/features/payments.md` exists with a Known Gotchas section listing "Retry on 503 can double-charge if not idempotent"
- **AND** the session work included fixing this issue and documenting the idempotency key mechanism
- **WHEN** `memory-update` runs the feature update path
- **THEN** the Known Gotchas section of `payments.md` is updated to mark the gotcha as resolved (e.g., with a resolution note and date)
- **AND** any new invariant (e.g., "All charge operations MUST include an idempotency key") is appended to Business Rules and Invariants
- **AND** a Decision Log entry is added if a significant implementation decision was made

#### Scenario: Session introduces a new domain invariant — persisted to feature file

- **GIVEN** `ai-context/features/auth.md` exists
- **AND** during the session, a new rule was established: "OAuth tokens MUST be rotated on every role-change event"
- **WHEN** `memory-update` runs the feature update path
- **THEN** the new rule is appended to the Business Rules and Invariants section of `auth.md`
- **AND** the date of the addition is recorded inline (e.g., as a `> Added: YYYY-MM-DD` note)

#### Scenario: memory-update does not create new feature files

- **GIVEN** the session touched a domain (`notifications`) for which no `ai-context/features/notifications.md` file exists
- **WHEN** `memory-update` runs the feature update path
- **THEN** `ai-context/features/notifications.md` is NOT created
- **AND** the update path is silently skipped for that domain
- **AND** the user MAY be informed in the summary that domain knowledge for `notifications` was not persisted due to absence of a feature file (informational, not an error)

#### Scenario: memory-update summary reports feature file updates

- **GIVEN** `memory-update` modifies `ai-context/features/auth.md`
- **WHEN** `memory-update` returns its summary
- **THEN** the modified feature file MUST be listed in the "Modified files" section of the summary
- **AND** a brief note of what was added MUST be included (e.g., "1 new invariant, 1 Decision Log entry")

#### Scenario: memory-update does not overwrite existing feature file content

- **GIVEN** `ai-context/features/payments.md` contains three existing business rules
- **WHEN** `memory-update` adds a fourth business rule
- **THEN** the original three rules MUST remain intact and unchanged
- **AND** only the new rule is appended
