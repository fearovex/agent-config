# Proposal: deprecate-commands-normalize-skills

Date: 2026-02-26
Status: Draft

## Intent

Remove the legacy `.claude/commands/` audit dimension from `project-audit` and normalize the entire scoring model around skills as the only supported extensibility mechanism, aligning with the official Claude Code standard.

## Motivation

Claude Code's official documentation confirms that `.claude/commands/` is the legacy approach and `.claude/skills/` is the current recommended standard. However, the current `project-audit` skill still allocates 10 points (Dimension 5) to auditing the commands directory, and Dimension 1 checks for a "Has Commands registry" entry in CLAUDE.md. This creates three concrete problems:

1. **False penalty**: Projects that correctly follow the modern standard (skills only, no commands/) lose 10 points in the audit score — they are penalized for doing the right thing.
2. **Misleading project-fix behavior**: When `/project-fix` runs on an audit that flagged a missing commands registry, it may add or repair legacy command files in the target project, actively pushing projects toward the deprecated approach.
3. **Registry drift in CLAUDE.md**: The "Has Commands registry" check in Dimension 1 causes `project-fix` to add a commands table to CLAUDE.md even in projects that should never have one, polluting the configuration.

The fix removes all audit pressure toward commands/, replaces the 10-point D5 with expanded skills coverage (D4 gains the freed points), and adds a passive INFO notice if a project still has a `commands/` directory (to guide migration, not penalize).

## Scope

### Included

- Remove Dimension 5 (Commands Quality) entirely from `project-audit/SKILL.md`
- Remove the "Has Commands registry" check row from Dimension 1's check table in `project-audit/SKILL.md`
- Remove the "Commands registry present" row from the Dimension 1 report format block in `project-audit/SKILL.md`
- Add a lightweight INFO check in `project-audit/SKILL.md`: if `.claude/commands/` exists in the target project, emit a LOW finding suggesting migration to skills (zero score penalty)
- Update the scoring table in `project-audit/SKILL.md`: move D5's 10 points to D4 (Skills, 10 pts → 20 pts); adjust score column headings accordingly; total stays 100
- Update the Detailed Scoring table at the bottom of `project-audit/SKILL.md` to reflect the redistribution
- Update the report format section in `project-audit/SKILL.md` to remove the D5 dimension block and its score row
- Update `project-fix/SKILL.md`: remove step 2.4 "Fix Commands registry" and add an explicit rule that the skill NEVER touches `.claude/commands/`
- Update `project-setup/SKILL.md`: confirm that no commands/ directory is created; add explicit note to the Rules section
- Update the global `CLAUDE.md` (at repo root and via install.sh to `~/.claude/CLAUDE.md`): remove any references to a "Commands registry" in the audit dimensions description or skill registry table

### Excluded (explicitly out of scope)

- Migrating any existing `.claude/commands/` files in test or user projects — this proposal only changes what the audit/fix/setup tools do, not the contents of other repos
- Changing how existing commands/ directories function for users who still have them — we do not break or delete anything, we only stop auditing and creating them
- Adding a migration wizard or automated conversion from commands/ to skills/ — that is a separate concern and not required for correctness
- Modifying any SDD phase skills (sdd-propose, sdd-spec, etc.) — they are unaffected
- Changing the CLAUDE.md template generated by `project-setup` to remove all mention of commands — `project-setup` never created a commands/ directory, only the CLAUDE.md template's "Active Skills" section is relevant, which already uses skills

## Proposed Approach

The change is purely textual: four SKILL.md files and the repo CLAUDE.md need specific sections removed or modified. No new files are created. No behavioral logic changes beyond removal of the commands audit path.

For the score redistribution: D4 (Skills Quality) currently caps at 10 points. After this change it caps at 20 points, compensating for the removed D5. The scoring rubric for D4 gains a second tier: registry accuracy + content depth (10 pts, existing) + recommended global skills coverage (10 pts, upgraded from informational to scored). This makes skills coverage more meaningful rather than simply inflating the existing checks.

For the INFO notice: added as a new sub-check in Dimension 1 or as a standalone note in the Phase A discovery script. If `d .claude/commands` returns 1, the audit emits a LOW finding: "Legacy `.claude/commands/` directory detected — migrate to `.claude/skills/` following the official Claude Code standard."

## Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Existing audit-report.md files in other projects reference D5 — project-fix may behave unexpectedly on stale reports | Low | Low | project-fix reads FIX_MANIFEST from the report; if D5 actions are present from a stale report, they will reference non-existent handlers and be silently skipped. Document this in known-issues.md |
| Score inflation — removing D5 and redistributing to D4 could make projects appear higher-scored without real improvement | Low | Medium | The D4 expansion (adding the global skills coverage sub-score) ensures the extra 10 points require real work to earn. The rubric is adjusted, not just moved |
| project-fix step 2.4 removal causes a regression if a project audit report contains `fix_commands_registry` actions | Low | Low | After this change, project-fix will not have a 2.4 handler. Any old FIX_MANIFEST entry of that type will be unrecognized and logged as "no handler — skipped". This is safe behavior |

## Rollback Plan

All changes are confined to four SKILL.md files and CLAUDE.md. To revert:

1. `git revert` the commit that contains this change — all four files return to their previous state in one operation.
2. Run `install.sh` to redeploy the reverted files from repo to `~/.claude/`.
3. No data migration is needed: no files in user projects are modified by this change.

If a partial rollback is needed (e.g., only revert project-audit but keep project-fix updated):
- `git checkout <previous-sha> -- skills/project-audit/SKILL.md`
- Run `install.sh`

## Dependencies

- No external dependencies. All changes are within this repo.
- `install.sh` must be run after apply to deploy updated skills to `~/.claude/`.
- A fresh `/project-audit` run on the claude-config repo itself (using Audiio V3 as the test target per config.yaml) is required for the verify-report.md.

## Success Criteria

- [ ] Running `/project-audit` on a project that has NO `.claude/commands/` directory scores the same or higher than before (no false penalty from a missing commands dimension)
- [ ] Running `/project-audit` on a project that HAS `.claude/commands/` emits exactly one LOW finding suggesting migration, with zero score penalty
- [ ] `project-fix` no longer creates or modifies any file under `.claude/commands/` in any project
- [ ] `project-setup` no longer creates a `.claude/commands/` directory in new projects
- [ ] The audit score table in `project-audit/SKILL.md` sums to 100 with D4 at 20 pts and no D5 row
- [ ] The global `CLAUDE.md` no longer mentions "Commands registry" as an audit dimension or CLAUDE.md requirement
- [ ] `/project-audit` run on this repo (claude-config) after apply scores >= the score before apply (currently tracked in `.claude/audit-report.md`)

Low (hours) — all changes are textual edits to existing SKILL.md files and CLAUDE.md. No new logic, no new files, no dependencies.
