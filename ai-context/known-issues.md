# Known Issues & Gotchas — claude-config

> Last updated: 2026-02-23

## rsync not available on Windows

`sync.sh` uses `rsync` which is not installed by default on Windows/Git Bash.

**Workaround**: Use `cp -r` manually:
```bash
cp ~/.claude/CLAUDE.md ~/claude-config/CLAUDE.md
cp ~/.claude/settings.json ~/claude-config/settings.json
cp -r ~/.claude/skills/. ~/claude-config/skills/
cp -r ~/.claude/memory/. ~/claude-config/memory/
cp -r ~/.claude/hooks/. ~/claude-config/hooks/
```

**Fix needed**: Update `sync.sh` to use `cp -r` with a delete step, or detect OS and use the right tool.

---

## install.sh is one-directional

`install.sh` copies FROM repo TO `~/.claude/`. If Claude modified skills during a session and `sync.sh` wasn't run, those changes are lost when `install.sh` is next executed.

**Rule**: Always `sync.sh` → `git commit` after any Claude session that modified skills or CLAUDE.md.

---

## settings.local.json is not tracked

`settings.local.json` at `~/.claude/` is NOT included in the repo (and should not be — it contains machine-specific or sensitive config generated by Claude Code). This means MCP server registrations may need to be re-run on new machines via `install.sh`.

---

## GITHUB_TOKEN dependency in install.sh

`install.sh` registers the GitHub MCP server with `${GITHUB_TOKEN}`. If this env var is not set when running `install.sh`, the MCP server registration will fail silently or with an error.

**Rule**: Define `GITHUB_TOKEN` as a system environment variable before running `install.sh`.

---

## Skills modified via Claude Code don't auto-sync

When a Claude Code session modifies `~/.claude/skills/` directly (not via the repo), those changes do NOT appear in `~/claude-config/` until `sync.sh` is run. This happened in the 2026-02-23 session (project-audit rewrite + project-fix creation were done in `~/.claude/` before sync).

---

## project-audit skill does not yet handle no-package.json projects

The current `project-audit` SKILL.md assumes a package.json exists for stack detection. For projects without one (like `claude-config` itself), the stack detection dimension will report a warning. The skill needs a fallback for non-Node projects.

---

## No automated test for skills

Skills have no automated tests. The only validation is running `/project:audit` and manually verifying skill behavior in a test project. A future improvement would be a `/skill:test` meta-tool.
